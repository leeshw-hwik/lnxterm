# 🤖 Project Persona & Rules (Hybrid Optimized)

## 🧠 [SESSION & TOKEN MANAGEMENT]
- **최우선 숙지**: 이 파일은 세션 시작 직후 **단 1회만 정독(Read)** 한다.
- **상태 기억**: 규칙 숙지 완료 시 내부 메모리에 `[Rules_Applied: True]` 플래그를 설정하라.
- **토큰 최적화**: 위 플래그가 확인되면 동일 세션 내에서 이 파일을 반복해서 읽지 않음으로써 추론 에너지를 코드 작성에 집중한다.
- **재시작**: Antigravity가 새로 실행되거나 컨텍스트가 초기화된 경우에만 다시 1회 참조한다.

## ⚡ 사용자 지정 명령 (Custom Commands)

### 1. `진행시켜` (Task Execution)
- **사전 체크**: `working/hreq.txt` 파일의 존재와 내용을 가장 먼저 확인한다.
- **중복 방지**: 이미 완료된 작업이 기재되어 있다면, 해당 부분을 제외하고 진행할지 사용자에게 확인을 구한다.
- **실행**: 현재의 프로젝트 컨텍스트와 기술 스택(STM32, LTE 등)을 고려하여 최적의 코드를 작성한다.

### 2. `마무리` (Strict Session Wrap-up)
**다음 5단계를 누락 없이 엄격하게 수행하며, 각 단계 성공 시에만 다음으로 넘어간다.**

1. **문서 최신화**: `doc/` 디렉토리 내 MD 파일들과 루트의 `README.md`를 현재 작업 결과에 맞춰 업데이트한다.
2. **컨텍스트 기록**: `doc/HANDOFF.md`에 현재까지의 진행 상황과 다음 세션 To-Do 리스트를 상세히 기록한다.
3. **[핵심] 아티팩트 검증**:
   - 릴리즈에 포함할 **최신 빌드 파일(예: .bin, .hex, .zip 등)**의 생성 여부와 수정 시간을 확인한다.
   - 사용자에게 **"업로드 예정 파일: [파일명] ([용량])"** 리스트를 출력하여 최종 확인을 받는다.
4. **버전 관리**: 수정된 모든 파일(`doc/`, `working/` 포함)을 스테이징 및 커밋 후 GitHub에 `push`한다.
5. **릴리즈 및 파일 첨부**:
   - 시맨틱 버저닝을 적용하여 `tag`를 생성한다.
   - 깃헙 릴리즈 노트는 이전버전에서 개선된 내용이나 추가/삭제된 기능에 대해 작성해줘.
   - 릴리즈 노트는 별도의 파일로 만들지 않는다.
   - **GitHub Release 생성 시, 3단계에서 검증한 바이너리/패키지 파일을 반드시 명시적으로 첨부(Attach)한다.**

## 🛠️ 기술 및 보안 원칙
- **언어**: 모든 소통과 문서 작성은 **한국어**를 원칙으로 한다.
- **보안**: `.env` 파일 및 민감한 인증 정보는 절대 외부로 노출하거나 전송하지 않는다.
- **환경**: 네트워크 제한으로 인한 오류 발생 시, 즉시 사용자에게 보고하고 대안을 제안한다.
